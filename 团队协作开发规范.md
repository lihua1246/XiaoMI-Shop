# 团队协作与 Claude Code 开发规范

本文档给小米商城前端实训项目使用。每位成员在用 Claude Code 开发前，先让 Claude Code 阅读本文档，并要求它按这里的规则执行。

## 1. 基本原则

本项目是纯静态前端项目，只使用 HTML、CSS、JavaScript。

- 不接入后端数据库。
- 不实现真实登录、支付、订单提交。
- 不安装新的前端框架。
- 不引入复杂构建工具。
- 不直接在 `main` 分支开发。
- 不让 Claude Code 自动提交、推送、删除文件或强制重置。

如果确实需要修改其他成员的文件，先在团队里说明原因，再让 Claude Code 操作。

## 3. 推荐分支名

以后不要使用 `codex/` 前缀。统一使用这些名字：

```text
feat/home-page
feat/products-page
feat/detail-page
feat/cart-page
feat/login-page
fix/page-style
docs/team-guideline
```

## 4. 给 Claude Code 的固定提示词

每次开始开发前，先复制下面这段给 Claude Code：

```text
请先阅读《团队协作与ClaudeCode开发规范.md》。
这是一个小米商城静态前端团队项目。
请不要直接在 main 分支开发。
请先确认当前分支；如果我还在 main，请提醒我创建自己的功能分支。
请只修改我负责范围内的文件，不要重构无关页面，不要引入新的框架或依赖。
不要自动执行 git commit、git push、git reset、删除文件等操作。
修改完成后，请列出改动文件、改动内容摘要、是否涉及其他成员文件，以及建议我检查的页面。
```

## 6. Claude Code 前提下的组员工作流

组员不需要记很多 Git 命令，只要按下面流程做。

### 第一次开始自己的页面

告诉 Claude Code：

```text
请帮我确认当前仓库状态。
如果现在在 main 分支，请基于最新 main 创建我的功能分支：feat/login-page。
创建后不要写代码，先告诉我当前分支是否正确。
```

把 `feat/login-page` 换成自己的分支名。

### 每天开始开发前

告诉 Claude Code：

```text
请帮我同步最新 main，并把最新 main 合并到我的功能分支。
如果有冲突，不要自动乱改，先告诉我冲突文件和冲突原因。
```

这一步的目的：让自己的分支拿到别人已经合进 `main` 的最新代码。

### 让 Claude Code 写功能时

告诉 Claude Code：

```text
我现在要开发我负责的页面。
请只修改我负责范围内的文件。
完成后不要提交 git，只输出改动文件列表、改动摘要和需要我检查的页面。
```

### 写完功能后

告诉 Claude Code：

```text
请帮我检查本次改动是否只包含我负责的文件。
请输出建议提交信息，但不要自动提交。
```

如果确认没问题，再由组员自己提交，或者让组长统一提交。

### main 又有新合并时

告诉 Claude Code：

```text
main 已经有新的合并提交。
请帮我把最新 main 同步到我的当前功能分支。
如果出现冲突，请只列出冲突位置和建议处理方式，不要直接覆盖文件。
```

## 7. 提交信息规范

提交信息统一使用：

```text
类型(范围): 本次提交做了什么
```

推荐示例：

```text
feat(home): 新增首页 banner 和入口按钮
feat(products): 完成商品列表渲染
style(products): 优化商品卡片布局
fix(cart): 修复购物车总价计算错误
docs(readme): 补充分工和运行说明
```

不要使用：

```text
修改
更新
111
完善页面
随便改一下
```

## 8. CSS 规范

`css/style.css` 是最容易冲突的文件，必须按页面分区维护。

推荐结构：

```css
/* ========== common ========== */

/* ========== home page ========== */

/* ========== products page ========== */

/* ========== detail page ========== */

/* ========== cart page ========== */

/* ========== login page ========== */
```

类名尽量带页面前缀：

```text
.home-banner
.home-entry
.products-grid
.products-card
.detail-gallery
.cart-item
.login-form
```

避免使用过于通用的类名：

```text
.box
.card
.title
.content
.left
.right
```

## 9. 组员交付说明模板

每位成员完成一个功能后，在群里按这个格式说明：

```text
分支：feat/login-page
建议提交：feat(login): 完成登录页表单布局
改动文件：login.html, css/style.css, js/main.js
完成内容：登录表单、按钮样式、基础输入校验
需要检查：登录页排版、导航跳转、移动端宽度
```

## 10. 组长合并前检查

组长合并前重点看：

- 分支名是否清楚。
- 提交信息是否能看懂。
- 是否只改了对应成员负责的文件。
- 是否误改了其他页面。
- 公共导航和页脚是否保持一致。
- 页面是否能直接打开。
- 移动端宽度下是否明显错位。

## 11. 最低质量要求

每次交付至少满足：

- 页面可以直接在浏览器打开。
- 顶部导航可以跳转到其他页面。
- 没有明显错位、重叠、按钮文字溢出。
- 控制台没有明显 JavaScript 报错。
- 文件命名清楚，不出现 `test.html`、`new.html`、`111.js` 这类文件。
- Claude Code 没有跨范围修改其他成员页面。
